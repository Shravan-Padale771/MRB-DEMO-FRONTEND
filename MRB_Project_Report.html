1.3 System Constraints

Security Constraints
• Role-Based Access: Separate portals for Admin and Student to ensure sensitive operations (like result publication) are restricted.
• Authentication: Future implementation must include secure login to protect personal student details and exam records.
• Data Privacy: Encryption of student data is required to comply with privacy standards for educational records.
Scalability Constraints
• High Volume: System must efficiently process thousands of student registrations and exam applications simultaneously.
• Concurrent Access: Architecture must support simultaneous access from hundreds of schools and thousands of students during result cycles.
• Multi-Region: Scalable organizational hierarchy (Region > Centre > School) to manage growth across Maharashtra.
Database Constraints
• Relational Storage: All core entities (Students, Exams, Results) must be stored in an RDBMS to maintain complex relationships.
• Data Integrity: Strict consistency required between applications and results using unique applicationId mappings.
• JSON Support: Flexible storage for dynamic paper structures within strictly typed relational tables.

2.4 Workflow / Process Flow

1

Admin Setup — Organizational Hierarchy

Admin creates Regions → Exam Centres (under regions) → Schools (under centres). This hierarchy determines where students and results are grouped.

2

Student Registration

Students register at /student/register providing personal details + school selection. The API call is POST /addStudent?schoolId={id}.

3

Exam Creation

Admin creates exams with full metadata: papers array, exam_details (identity, schedule, rules, administrative info), fees, and date windows.

4

Application Period

Students browse active exams and apply via POST /fill-form. Application status starts as PENDING. Admin can review via Application Manager.

5

Result Publication

Admin enters paper-wise marks in Result Publisher. System calculates percentage, determines Pass/Fail (≥40%), and publishes via POST /addExamResult.

6

Student Result Viewing

Student logs in by Student ID, views results via Result Viewer, and downloads an official Marksheet (Anksoochi) rendered in the browser.

Proposed Future Enhancements (PROPOSED ALGORITHMS)

The following features and algorithms are proposed for the next phase of development to further automate the board's operations and enhance system security.

1. Automated Seat Allocation
A constraint-satisfaction algorithm can be implemented to automatically assign students to exam centers based on geographic proximity and center seat capacity, ensuring an even distribution and reducing travel for students.
2. Verifiable Marksheet Hashing
To prevent fraudulent marksheets, a SHA-256 Hashing algorithm is proposed. It would generate a unique digital fingerprint for every student's result, embeddable as a QR code for instant authentication by employers or institutions.
3. Statistical Anomaly Detection
A pattern-matching algorithm to analyze result sets for outliers. It can flag centers with suspiciously high pass rates or identical answer patterns for manual review, acting as an automated anti-cheating measure.
4. Intelligent Exam Scheduling
A heuristic-based scheduling algorithm to manage complex timetables for multiple exam levels (Prathamik, Prabodh, Pravin), ensuring zero time conflicts for students appearing in multiple papers.
5. Weighted Performance Analytics: Comparative ranking algorithms to provide statewide percentile scores and school-wise performance benchmarks.

5.3 N+1 Query Problem Resolution

During initial load testing of the /getAllApplications endpoint, a classic ORM N+1 query problem was discovered. Fetching a list of applications inadvertently triggered separate database queries to fetch the associated Student and Exam entities for every single application in the list.

This resulted in massive overhead (e.g., 100 applications = 1 query for apps + 100 queries for students + 100 queries for exams = 201 total queries), leading to response times of over 6 seconds.

The Solution: The backend was refactored to implement rigorous Data Transfer Objects (DTOs) combined with custom JPQL JOIN FETCH queries. Instead of returning deeply nested Hibernate proxy entities that trigger lazy loading, the API now projects exactly the required fields into a flat ApplicationDTO.

// Example of the flat, efficient DTO output (output_json/GET_getAllApplications.json) { "applicationId": 1, "examNo": 5, "examName": "Rashtrabhasha Prabodh Pariksha (May 2026)", "studentId": 1, "studentName": "Aarav Sharma", "status": "PENDING" }
Conclusion: Implementing caching using Redis/Spring Cache resulted in up to ~268x faster response times for heavy payloads like applications and results. Furthermore, the integration of DTOs permanently resolved the N+1 query vulnerability, ensuring reliable list endpoint delivery in ~20-40ms, down from 1-6 seconds.
